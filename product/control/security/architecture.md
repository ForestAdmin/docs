## Data privacy

### Private by design

Forest Admin implements a **privacy-first architecture** where your sensitive data flows directly between your Agent and user browsers, never passing through Forest Admin servers.

![Forest Admin security architecture](../../../reference/assets/security-privacy-architecture.png)

{% hint style="info" %}
**Your data never transits through our servers.** When users access Forest Admin, their browser connects directly to your Agent to retrieve data. Forest Admin servers only provide the UI layout and configuration.
{% endhint %}

### No third-party vendor tracking

Forest Admin does not include tracking pixels or third-party analytics that could monitor your data or activity. Your information remains private within your infrastructure.

## Security

### Tokens

Forest Admin uses two environment secrets to authenticate requests:

#### FOREST_ENV_SECRET

Authenticates requests between your Agent and Forest Admin servers.

- Links your Agent to your Forest Admin project
- Required for all architectures (Cloud, Self-Hosted, On-Premise)
- Generated by Forest Admin during onboarding

```bash
FOREST_ENV_SECRET=1234567890abcdef1234567890abcdef1234567890abcdef
```

{% hint style="warning" %}
Never commit `FOREST_ENV_SECRET` to version control. Always use environment variables or secret management tools.
{% endhint %}

#### FOREST_AUTH_SECRET

Authenticates requests between user browsers and your Agent (Self-Hosted and On-Premise only).

- Signs JWT tokens for user authentication
- Generated by you (not Forest Admin)
- Should be at least 32 characters long

```bash
# Generate a secure secret
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

# Or use openssl
openssl rand -hex 32
```

{% hint style="info" %}
**Cloud architecture:** `FOREST_AUTH_SECRET` is not needed because authentication is handled by Forest Admin servers. Your data still flows directly from your Agent to browsers.
{% endhint %}

### IP whitelisting

Restrict access to Forest Admin by whitelisting specific IP addresses or ranges. This prevents unauthorized access even if credentials are compromised.

See [IP whitelisting](ip-whitelisting.md) for configuration details.

### DMZ & VPN

Deploy your Agent behind a VPN or in a DMZ (Demilitarized Zone) for additional network security. This ensures that even with valid credentials, attackers need network access to reach your Agent.

### Credentials

Your database credentials and sensitive configuration never leave your infrastructure. Forest Admin servers never have access to:

- Database credentials
- `FOREST_AUTH_SECRET`
- Your actual business data

All sensitive data remains within your control.
