A **Workspace** is a custom application you build inside Forest Admin, tailored to a specific workflow. Instead of navigating between multiple collections, operators get a single focused interface that shows exactly what they need and lets them take action immediately.

Think of a Workspace as the difference between a generic spreadsheet and a purpose-built tool: same data, dramatically better experience.

## Why use workspaces

The default "Data" tab gives operators access to all your collections, but it's generic. Workspaces let you:

- **Focus** — show only the data relevant to a specific task
- **Contextualize** — display related information side by side without navigating away
- **Act faster** — put action buttons directly next to the data that triggers them
- **Guide** — add text, sections, and structure to make workflows self-explanatory

Workspaces are built with a **drag-and-drop editor** — no code required for most use cases.

## Workspace components

A Workspace is composed of **components** that you drag onto a canvas and configure:

| Component | Purpose |
|-----------|---------|
| **Text** | Headers, instructions, labels with optional dynamic templating |
| **Section** | Groups related components under a collapsible heading |
| **Tabs** | Splits the workspace into tabbed panels |
| **Divider** | Visual separator between areas |
| **Collection** | Displays a filtered list of records from any collection |
| **Field** | Shows the value of a specific field from a selected record |
| **Chart** | Embeds a chart or metric (single value, distribution, etc.) |
| **Button** | Triggers a Smart Action on a selected record |
| **Search** | Lets operators search for a record from a collection |
| **Dropdown** | A selector that filters connected components dynamically |
| **Toggle** | A boolean switch that controls connected components |
| **Link** | A link to a record, external URL, or another workspace |

## Templating: making workspaces dynamic

The real power of Workspaces comes from **Templating** — the ability to reference values from other components dynamically. Type `{{` anywhere in a text, filter, or link to see available variables.

For example:
- A **Text** component can display `Welcome, {{currentUser.name}}`
- A **Collection** component can filter records where `company_id = {{collection1.selectedRecord.id}}`
- A **Link** can redirect to `https://app.example.com/users/{{collection1.selectedRecord.id}}`

When an operator selects a row in `collection1`, all connected components update instantly.

## Building a workspace

**Step 1 — Create a new workspace**

Click the **🧩** icon in the left sidebar, then click "Create your first Workspace". Give it a descriptive name that reflects the workflow (e.g. "KYC Review", "Fraud Queue").

**Step 2 — Add a Collection component**

Drag a **Collection** from the sidebar onto the canvas. Select which collection to display and optionally apply a segment (e.g. "Pending applications"). Set the row click behavior to **"Select a record"** to enable dynamic updates.

**Step 3 — Add context with Field components**

Drag **Field** components onto the canvas. Link them to your Collection and select which fields to display. When an operator clicks a row, these fields update to show the selected record's values.

**Step 4 — Add action buttons**

Drag a **Button** and link it to your Collection and a Smart Action. Now operators can click a row and immediately trigger the right action — no need to open the record detail.

**Step 5 — Add charts and text for context**

Use **Chart** components to show aggregated metrics (e.g. "Pending applications today"). Use **Text** components with Templating to show dynamic labels and guide operators through the workflow.

**Step 6 — Exit the builder and test it**

Click "Exit builder" to switch to the operator view. Test the workflow by clicking rows and triggering actions.

## Sharing workspaces

Workspaces are available to all operators in your project by default. Control access using [Roles & Permissions](/product/control/roles-permissions) — you can limit Workspace visibility to specific teams.

Operators can also share a Workspace URL with a pre-applied filter state, making it easy to hand off specific work items.
