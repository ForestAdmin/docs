## What you'll need

| Requirement | Notes |
| --- | --- |
| HTTPS-accessible agent URL | Forest's UI talks to your agent over HTTPS. Localhost and HTTP are not allowed for production. |
| Environment variables | `FOREST_ENV_SECRET`, `FOREST_AUTH_SECRET`, `DATABASE_URL`. Plus `NODE_ENV=production` (Node) or `RAILS_ENV=production` (Ruby). |
| `.forestadmin-schema.json` | Generated automatically in development. Must be committed and deployed for production environments. |
| Forest CLI | Used to push layout changes between environments. See [CLI reference](/reference/cli/overview). |

## Deployment workflow

The full path from local development to production:

<Steps>
  <Step title="Develop locally">
    Run your agent on `localhost:3310`. Connect to your Forest development environment. Use the Layout Editor in Forest to make UI changes — these are saved on a branch.
  </Step>
  <Step title="Deploy your agent code to production">
    Push your code to your hosting platform ([AWS](/guides/deployment/platforms/aws), [Heroku](/guides/deployment/platforms/heroku), or your own infrastructure). The agent must be reachable via HTTPS.
  </Step>
  <Step title="Create the Forest production environment">
    In Forest → Project Settings → Environments → Add. Enter your agent's URL. Set the environment variables Forest generates on your server.
  </Step>
  <Step title="Push layout changes to staging (optional)">
    Run `forest push` to push your branch to a remote (staging) environment.
  </Step>
  <Step title="Deploy layout changes to production">
    Run `forest deploy` (or click "Deploy to production" in the staging banner). Always deploy backend code first, then run `forest deploy`.
  </Step>
</Steps>

<Warning>
  `forest deploy` is irreversible. Always deploy your backend code changes (datasource configuration, Smart Actions, computed fields) **before** running `forest deploy`. The layout deployment expects the schema to match.
</Warning>

## Environment hierarchy

Forest separates two concepts:

- **Environments** — your local dev, staging, production. Each has its own database, agent URL, and `FOREST_ENV_SECRET`.
- **Branches** — layout changes in progress on top of an environment. Branches only exist in development; they're pushed to a remote environment with `forest push` and deployed with `forest deploy`.

A typical hierarchy:

```
development → staging → production
```

For larger teams: `development → test → staging → preprod → production`. See [Environments](/guides/deployment/environments).

## Pre-launch checklist

<AccordionGroup>
  <Accordion title="Agent deployed and HTTPS-accessible">
    Your agent's URL must respond on HTTPS, with no trailing slash. Test with `curl https://your-agent.com/forest` — should return Forest metadata.
  </Accordion>
  <Accordion title="Environment variables set">
    `FOREST_ENV_SECRET`, `FOREST_AUTH_SECRET`, `DATABASE_URL`, plus `NODE_ENV=production` or `RAILS_ENV=production`. All set on the production server.
  </Accordion>
  <Accordion title="Schema file committed">
    `.forestadmin-schema.json` is generated in development on agent startup. Commit it to version control and deploy with your code.
  </Accordion>
  <Accordion title="Production environment created in Forest">
    Project Settings → Environments → "Add a new environment". Enter the agent URL.
  </Accordion>
  <Accordion title="CRUD operations verified in production">
    Browse a collection. Open a record. Edit a field. Confirm the change persists. If anything fails, check agent logs and database connectivity.
  </Accordion>
  <Accordion title="Smart Actions tested">
    For each Smart Action, run it against a non-destructive test record and confirm the expected outcome.
  </Accordion>
  <Accordion title="Permissions configured">
    Project Settings → Roles. Confirm each role has the right access to collections, fields, and actions before exposing the back-office to your team.
  </Accordion>
</AccordionGroup>
