The `.forestadmin-schema.json` file is the canonical description of your agent's data model, the collections it exposes, the fields and their types, the relationships, and the customizations applied. It's auto-generated, versioned with your code, and read by the Forest API to render the UI for each environment.

<Note>
  The filename is `.forestadmin-schema.json` (the legacy naming is preserved to avoid breaking existing deployments). Treat it as a Forest-managed file.
</Note>

For the day-to-day workflow of keeping the schema in sync when your database changes, see [Schema updates](/product/process/advanced-concepts/developer-workflow/schema-updates).

## When the file is generated

In **development environments only**, the agent regenerates `.forestadmin-schema.json` on every startup. It reflects:

- The state of your data sources (collections, fields, types, relationships)
- Your agent's customizations (actions, computed fields, segments, hooks)

In Node.js, the agent decides whether to regenerate based on the `isProduction` option passed to `createAgent`. In Ruby, generation is tied to the Rails environment, and you can also generate the file explicitly with a rake task.

<CodeGroup>
```typescript Node.js
const agent = createAgent({
  authSecret: process.env.FOREST_AUTH_SECRET,
  envSecret: process.env.FOREST_ENV_SECRET,
  isProduction: process.env.NODE_ENV === 'production',
});
```

```bash Ruby
# In development, the schema is regenerated when the server boots.
# In production, the agent uses the committed file and never regenerates it.
# Ruby v2 can also generate it explicitly, without booting the server:
rails forest_admin:schema:generate

# Add debug=true to troubleshoot generation:
rails forest_admin:schema:generate debug=true
```
</CodeGroup>

| Environment | Behavior |
| --- | --- |
| Development | Regenerates `.forestadmin-schema.json` on every restart |
| Remote / production | Never regenerates, uses the schema file deployed alongside the code |

## Why versioning matters

In **remote and production environments**, the agent does NOT regenerate the schema file. It uses the version that was deployed with your code. This means:

- You must commit `.forestadmin-schema.json` to version control
- You must deploy it alongside your agent code to every remote environment
- The file is the contract between your agent and the Forest API for that environment

If you change your data sources or customizations and don't redeploy the schema file, the production environment will keep serving the old schema, your changes won't show up in the UI.

<Warning>
  Do not edit `.forestadmin-schema.json` manually. Wrong syntax will break the UI.
</Warning>

### Why it isn't regenerated at runtime

Regenerating the schema live instead of shipping the committed file would be unstable:

- **It can break your layout.** Your [layout](/guides/best-practices/layout-maintenance) (fields shown, column order, segments, Smart Actions, custom views) is built on top of the schema and references its collections and fields by name. A schema that shifts unexpectedly between boots can reset or break that layout.
- **It flip-flops during deploys.** Production runs several instances, and during a rolling deploy old and new versions run side by side. If each regenerates and posts its own schema, they overwrite each other on the Forest API and the UI flips between versions. A committed file means every instance posts the same schema.
- **It couples startup to the database.** Generating means introspecting your tables, so the agent would need a live, migrated database just to boot.
- **It slows cold starts.** That introspection would run on every instance startup.

So the schema is generated once in development, committed, and deployed frozen. The [CI check below](#catch-an-uncommitted-schema-in-ci) just verifies the committed file stays up to date.

## What versioning the schema gives you

Beyond keeping environments in sync, committing the schema file lets you:

- **Review schema changes in pull requests**, diffs show every collection, field, and relationship change before merge
- **Roll back schema regressions**, if a bad change breaks production, revert the commit
- **Audit historical schemas**, Git history shows the data model state at any past point

## Catch an uncommitted schema in CI

If a developer changes a data source or customization but forgets to commit `.forestadmin-schema.json`, the deployed agent serves a stale schema and the change never reaches the UI.

Add a CI step that **regenerates the schema and fails if it differs from what's committed**. A red build then means "you forgot to commit your schema".

The agent writes `.forestadmin-schema.json` on boot regardless of language, but booting also contacts Forest, so you can't just run it offline to regenerate the file. In CI, boot it against a local Forest sandbox (`createForestServerSandbox`), which mocks those calls, generate the schema to a temporary path, and compare it against the committed file. See the full Node.js example in [Check the committed schema is up to date (CI)](/product/process/advanced-concepts/testing#check-the-committed-schema-is-up-to-date-ci).

<Tip>
  **Ruby (v2)** can skip the sandbox: the `rails forest_admin:schema:generate` rake task writes the file directly, without booting the server or contacting Forest. It introspects your tables, so the database must be up and migrated first.

  ```bash
  rails db:create db:migrate
  rails forest_admin:schema:generate
  git diff --exit-code .forestadmin-schema.json
  ```
</Tip>
